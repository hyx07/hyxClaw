# 阶段 5 规格文档：UI 可读性与交互优化

## 1. 目标

阶段 5 的核心目标是：在不重写前端架构的前提下，让当前聊天页面更清晰、更易读、更像一个完整产品界面。

本阶段不引入 React / Vue / Svelte，不引入 `shadcn/ui`，继续基于现有纯 HTML / CSS / JS 页面迭代。

重点不是“换组件库”，而是：

1. 提升 assistant 回复内容的 Markdown 可读性
2. 优化 Thinking / 工具调用块的视觉层级
3. 调整消息区、右侧栏、底部输入区的整体观感
4. 保持当前页面布局和功能入口基本不变

---

## 2. 范围

### 2.1 本阶段实现

- Assistant 消息支持 Markdown 渲染
- Markdown 渲染需要安全处理，不能直接执行模型输出中的危险 HTML
- Thinking 块视觉优化
- 工具调用块视觉优化
- 消息布局优化：user 与 assistant 消息都靠左展示；user 保留气泡，assistant 改为正文流展示
- 移除消息气泡中的 `AI` / `你` 标签
- 右侧 rail 保留并加宽，按钮带文字说明
- `Token Stats` 从顶部区域移动到右侧 rail
- 底部输入区改为上下两层：
  - 上层：文本输入框
  - 下层：状态与功能区
- 底部状态与功能区仅保留现有功能：
  - `Compact`
  - token 计数摘要
  - `Thinking`
  - 发送按钮

### 2.2 本阶段不实现

- 暗色模式
- 代码高亮
- 代码块复制按钮
- 会话列表搜索
- 消息时间戳
- React / Vue / Svelte 等前端框架迁移
- `shadcn/ui`
- 前端构建链重构
- 新增当前后端不存在的功能入口

---

## 3. 设计原则

### 3.1 保持当前布局

页面整体仍然保持当前结构：

- 左侧：会话列表
- 中间：聊天内容
- 右侧：rail 功能栏
- 底部：输入区

不做大范围布局重排，不把页面改成全新的产品形态。

### 3.2 内容优先

聊天页面的主要价值是内容可读性。本阶段优先优化：

- Markdown 文本结构
- 段落、列表、表格、引用、代码块的阅读体验
- Thinking 与工具调用的层级区分
- user / assistant 消息的视觉区分

### 3.3 低侵入实现

实现应尽量局限在现有前端页面中：

- 继续使用 `src/server/public/index.html`
- 继续使用原生 DOM 操作
- 不改变 WebSocket / HTTP 主协议
- 不引入复杂状态管理
- 不为了视觉优化重写现有业务逻辑

### 3.4 安全优先

Assistant Markdown 渲染必须考虑安全性。

模型输出不能无保护地直接插入 `innerHTML`。如使用 Markdown 渲染库，必须明确禁用危险 HTML，或在渲染后进行 HTML 清洗。

## 4. 功能规格

## 4.1 Assistant 消息 Markdown 渲染

### 4.1.1 目标

将 assistant 的纯文本回复渲染为 Markdown 富文本，提高以下内容的可读性：

- 标题
- 加粗 / 斜体
- 有序 / 无序列表
- 引用块
- 行内代码
- 代码块
- 表格
- 分隔线

### 4.1.2 范围约束

- 仅对 `assistant` 消息启用 Markdown 渲染
- `user` 消息保持纯文本展示
- 工具结果块保持纯文本展示，不做 Markdown 渲染
- Thinking 块保持纯文本展示，不做 Markdown 渲染
- 不做代码高亮
- 不做代码块复制按钮

### 4.1.3 渲染策略

- Assistant 消息在落盘、传输、历史保存时仍然保存原始文本
- Markdown 只发生在前端展示层
- 流式输出过程中可以先按纯文本追加
- 在 `chatEnd` 或历史会话加载时，对 assistant 消息整体进行一次 Markdown 渲染
- 避免每个 chunk 到来时都做完整 Markdown 重渲染

### 4.1.4 安全要求

必须满足以下任一策略：

1. 使用 Markdown 渲染库时显式禁用危险 HTML
2. 或在 Markdown 渲染后增加 HTML 清洗步骤

最低要求：

- 不允许模型输出中的原始 `<script>` 执行
- 不允许 `onerror=`、`onclick=` 等事件属性执行
- 外链默认带 `rel="noopener noreferrer"`
- 外链如使用 `target="_blank"`，必须同时使用安全 `rel`

### 4.1.5 样式要求

为 Markdown 内容增加专用样式容器，例如：

```html
<div class="message-bubble markdown-body">...</div>
```

需要优化以下元素：

- `h1` ~ `h4`
- `p`
- `ul` / `ol`
- `blockquote`
- `code`
- `pre`
- `table`
- `hr`

样式要求：

- 最大行宽适中，避免长段落过宽
- 段落间距清晰
- 列表缩进自然
- 表格边框和表头层级清楚
- 代码块与正文有明显区分，但不需要语法高亮

---

## 4.2 消息布局优化

### 4.2.1 目标

当前“AI 在左、你在右”的布局不符合本项目期望。本阶段改为所有消息都靠左展示；user 消息保留气泡，assistant 消息不再使用气泡外框，而是像正文一样直接写在消息区背景上。

### 4.2.2 具体要求

- `assistant` 消息靠左
- `user` 消息也靠左
- 移除消息中的 `AI` 标签
- 移除消息中的 `你` 标签
- user 使用气泡背景
- assistant 不使用气泡背景、边框、圆角和阴影
- 不通过左右位置区分消息来源
- 不改变消息顺序

### 4.2.3 视觉建议

- User 气泡可使用稍暖的浅米橙色
- Assistant 消息像正文流一样展示在背景上，只保留 Markdown 内容本身的层级
- 两者都应与当前 Solarized 风格协调
- 消息宽度不宜铺满整行，保持当前最大宽度约束

---

## 4.3 Thinking 与工具调用块展示优化

### 4.3.1 目标

Thinking 块和工具调用块功能已经存在，本阶段只优化展示，不改变协议和数据结构。

### 4.3.2 Thinking 块

Thinking 块标题统一为：

```text
Thinking
```

标题前使用黑色圆点作为简洁标识，不使用 emoji 或复杂图标。

要求：

- 标题区更明确
- 默认折叠，仅显示标题行
- 标题后显示小三角 `▶`，点击后展开内容
- 展开时小三角旋转，表示当前为展开状态
- 展开内容包裹在稍深的浅米色内容框中，与最终 assistant 回复区分
- Thinking 内容保持纯文本显示
- 不增加复杂动画

### 4.3.3 工具调用块

工具调用标题按工具类型显示为更短的英文文案。

当前确认的示例：

- 读取类工具：`Read`
- 写入类工具：`Write`

标题前同样使用圆点作为简洁标识。

工具调用块仍保留：

- 工具类型标题
- 路径标签
- 执行状态，但不显示 `完成` / `失败` 文案胶囊
- 工具结果内容

工具调用块交互要求：

- 默认折叠，仅显示标题行
- 标题行显示：状态圆点、工具类型、路径标签、小三角
- 小三角放在路径标签后面
- 点击标题行后展开具体工具结果
- 展开的工具结果包裹在稍深的浅米色内容框中，与 assistant 正文区分

### 4.3.4 状态展示

如果后端已经提供工具调用成功 / 失败状态，则前端应继续展示对应状态。

要求：

- 成功状态与失败状态需要有视觉区分
- 工具调用开始时，标题前圆点为黑色
- 工具调用成功后，标题前圆点变为绿色
- 工具调用失败后，标题前圆点变为红色
- 不显示额外的 `完成` / `失败` 状态胶囊
- 不新增后端没有提供的状态
- 不做工具调用时间线视图

### 4.3.5 非目标

- 不改变单条工具调用的数据结构
- 不改变工具调用协议
- 不做复杂图标系统
- 不引入第三方 icon 库
- 不做复杂动画

---

## 4.4 右侧 Rail 与 Token Stats

### 4.4.1 目标

右侧 rail 保持现有位置，但宽度需要比早期版本更宽，按钮需要同时包含图标和文字说明。

### 4.4.2 右侧 Rail

要求：

- 右侧 rail 宽度加宽，约为早期窄图标栏的 1.5 倍
- rail 按钮不只显示图标，也显示文字
- 当前确认保留 `Token Stats`
- 不在 rail 中加入当前不存在的新功能

### 4.4.3 Token Stats 按钮

按钮文案为：

```text
Token Stats
```

要求：

- 放在右侧 rail 中
- 有轻微按钮感，但不要过强的 3D 效果
- hover 时需要有可点击反馈
- 风格与页面整体米色系保持一致

### 4.4.4 Token Stats 弹窗

点击 `Token Stats` 后打开 token 统计弹窗。

弹窗要求：

- 弹窗内容以表格形式展示 provider / model 聚合统计
- 关闭按钮使用 `×`
- 关闭按钮 hover 时需要有反馈，例如：
  - 背景略变暖
  - 边框加深
  - `×` 颜色变化
- 关闭按钮 active 时可以有轻微下压效果
- 点击遮罩区域可以关闭弹窗

---

## 4.5 底部输入区

### 4.5.1 目标

底部输入区改为上下两层，让输入与状态 / 功能操作分离。

结构：

1. 上层：输入文字框
2. 下层：状态显示和功能按钮

### 4.5.2 功能范围

下层只保留当前已有或已实现的功能：

- `Compact`
- token 计数摘要
- `Thinking`
- 发送按钮

不加入其他未实现功能入口。

### 4.5.3 左侧区域

左侧显示：

```text
Compact
Input: 12.5k Output: 1.9k Total: 14.4k
```

要求：

- `Compact` 是按钮
- token 统计是纯状态文本
- token 数字使用 `k` 作为单位
- 文案格式使用首字母大写：

```text
Input: 12.5k Output: 1.9k Total: 14.4k
```

### 4.5.4 右侧区域

右侧显示：

- `Thinking`
- 发送按钮

要求：

- `Thinking` 放在输入区下层右侧
- 发送按钮仍在最右侧
- 不改变发送功能

### 4.5.5 输入区颜色

当前确认的设计：

- 输入框上层与下方工具层使用同一个背景色
- 颜色使用更暖的米黄色，避免偏青灰
- 当前 mock 中使用：

```css
#f5ecd8
```

要求：

- 输入框和下方工具栏颜色必须一致
- 输入框和工具栏之间不需要横线
- 输入框 focus 时不出现明显底部分割线
- 不再随意改变底部输入框高度

---

## 4.6 顶部与整体样式统一

### 4.6.1 目标

统一当前页面已有操作元素的视觉风格，使页面不再像“功能逐步堆上去的按钮集合”。

### 4.6.2 涉及元素

- 左侧会话列表
- 中间消息区
- 右侧 rail
- `Token Stats` 按钮
- `Compact` 按钮
- `Thinking` 按钮
- 发送按钮
- Token Stats 弹窗

### 4.6.3 样式要求

- 圆角、边框、hover、active、disabled 状态尽量统一
- 按钮可以有轻微层次，但不要强 3D 效果
- hover 反馈要可感知，但不要夸张
- 整体继续沿用当前 Solarized / 米色系方向
- 避免 emoji 图标
- Thinking 标识使用黑色圆点
- 工具调用标识使用状态圆点：黑色表示调用中，绿色表示成功，红色表示失败

## 5. 技术方案

## 5.1 前端依赖

本阶段允许引入轻量级 Markdown 渲染库，但不引入前端框架。

推荐方式：

- 使用 npm 依赖，并通过现有项目构建链管理版本

可选方式：

- 使用 CDN 版本快速接入

选择依赖时优先考虑：

- 体积小
- 配置简单
- 可禁用危险 HTML
- 与当前纯 HTML / CSS / JS 页面集成成本低

如需要 HTML 清洗，可选择小型 sanitizer，或使用 Markdown 库自身的安全配置能力。

---

## 5.2 渲染策略

### 流式输出中

- assistant 流式输出阶段仍按纯文本追加
- 避免每个 chunk 都完整重渲染 Markdown
- 保持现有 streaming 体验稳定

### 流式结束后

- 使用完整 assistant 文本统一渲染一次 Markdown
- 渲染后写入 assistant 正文展示层

### 会话加载时

- 对历史 assistant 消息进行 Markdown 渲染
- 对 user 消息保持纯文本展示
- 对 Thinking / 工具调用内容保持纯文本展示

---

## 5.3 建议涉及文件

| 文件 | 改动类型 |
| --- | --- |
| `src/server/public/index.html` | 主要实现文件：Markdown 渲染、消息布局、Thinking / 工具块、输入区、右侧 rail 样式 |
| `package.json` | 如采用 npm 依赖，新增 Markdown / sanitizer 相关依赖 |

| `scripts/stage5_ui_mock.html` | 当前阶段 5 设计 mock，可作为实现参考 |

## 6. 验收标准

- Assistant 回复中的 Markdown 能正确渲染标题、列表、引用、代码块、表格等常见结构
- Markdown 渲染不执行危险 HTML / 脚本
- User 消息仍保持纯文本显示
- User 与 assistant 消息都靠左展示
- 消息气泡中不再显示 `AI` / `你`
- User 消息保留气泡背景
- Assistant 消息不使用气泡外框，像正文一样直接展示在消息区背景上
- Thinking 块标题显示为 `Thinking`
- 工具读取类标题显示为 `Read`
- 工具写入类标题显示为 `Write`
- Thinking / 工具块默认折叠，点击标题行后展开
- Thinking / 工具块标题行显示小三角 `▶`
- 工具块小三角放在路径标签后面
- Thinking 展开内容和工具结果内容使用稍深的浅米色内容框
- Thinking 标题前使用黑色圆点
- 工具调用中圆点为黑色，成功为绿色，失败为红色
- 工具调用不显示 `完成` / `失败` 状态胶囊
- 右侧 rail 加宽，按钮显示文字说明
- `Token Stats` 位于右侧 rail
- `Token Stats` 弹窗关闭按钮 hover / active 有反馈
- 底部输入区分为输入框和下方状态 / 功能区
- 底部下层只包含 `Compact`、token 计数、`Thinking`、发送按钮
- token 计数格式为 `Input: 12.5k Output: 1.9k Total: 14.4k`
- 输入框和下方工具栏背景色一致，且不偏青
- 不引入 React / `shadcn/ui`
- 不实现暗色模式、代码高亮、代码复制、会话搜索、消息时间戳
- `pnpm build` 通过
- `pnpm test --run` 通过

---

## 7. 实施顺序建议

1. 先完成 Markdown 渲染与安全处理
2. 再调整 assistant / user 消息布局与气泡样式
3. 优化 Thinking / 工具调用块标题与状态展示
4. 调整右侧 rail 与 `Token Stats` 弹窗
5. 调整底部输入区两层结构和 token 摘要
6. 最后统一按钮 hover / active / disabled 样式

这样即使中途停止，也能优先交付最有价值的“内容可读性提升”。
