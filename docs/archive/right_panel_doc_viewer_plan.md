# 右栏文档查看器计划

## 目标

在现有右侧栏加入一个文档查看组件，满足这些需求：

- 右栏可调节宽度，便于根据文件内容大小调整阅读区域
- 不做搜索，改为文件树式选择
- 顶层固定展示两个入口：`knowledge_base`、`inputs`
- 文件选择器固定 3 列
- 继续深入目录时，复用最后一列，不再继续新增列
- 支持回退到上一级
- 下方提供文件预览区
- 第一列和第二列默认都不自动选中任何条目
- 默认不自动选择任何文件
- Markdown 文件按现有渲染风格展示
- 提供手动刷新按钮，不做自动更新
- 前端发送用户消息时，附带当前预览文件的相对路径
- 如果用户在预览区选中了文字，也附带该选中文本
- 选中文本在前端保存为持久状态，不依赖发送瞬间的浏览器原生选区
- 不做正文内持久可视高亮，只做顶部状态提示

## 预期交互

右栏拆成两块：

- 上方是文件选择器
- 下方是文件预览区

最终 UI 草图：

```text
+------------------------------------------------------+
| 文档侧栏                                   [可拖拽] |
|------------------------------------------------------|
| [ Token Stats ]   [ Knowledge Base ]                 |
|------------------------------------------------------|
| 文件选择器                                           |
|------------------------------------------------------|
| +----------------+ +----------------+ +-----------+ |
| | 根目录         | | 中间列         | |← 返回 右侧列| |
| |----------------| |----------------| |-----------| |
| | knowledge_base | | docs/          | | guide.md  | |
| | inputs         | | archive/       | | api.md    | |
| |                | | index.md       | | notes/    | |
| |                | | prompt.md      | | more/     | |
| +----------------+ +----------------+ +-----------+ |
|                                                      |
| 说明：固定 3 列。继续深入时，右侧列复用为下一层内容。   |
|                                                      |
| 已选中第 x 行到第 y 行                     [清除][刷新] |
| “这里显示选中文字缩略内容……”                         |
|------------------------------------------------------|
| 文件预览区                                           |
|------------------------------------------------------|
| # 标题                                               |
|                                                      |
| 这里显示 Markdown 渲染后的内容……                     |
|                                                      |
| - 列表项 A                                           |
| - 列表项 B                                           |
|                                                      |
| ```ts                                                |
| const x = 1;                                         |
| ```                                                  |
|                                                      |
+------------------------------------------------------+
```

文件选择器交互方式：

1. 第一列固定显示 `knowledge_base` 和 `inputs`
2. 默认状态下第一列不自动选中任何条目
3. 点击第一列某一项后，第二列显示该目录下的内容，但第二列也不自动选中任何条目
4. 点击第二列中的目录后，第三列显示该目录下的内容，但不自动选择文件
5. 如果继续进入更深层目录，仍然复用第三列显示下一层内容，但不自动选择文件
6. 除了返回按钮，也允许直接点击上层列重新切换当前路径
7. 点击文件后，在下方预览区显示内容

预览区交互方式：

1. 提供 `刷新` 按钮
2. 点击刷新后重新加载文档结构和当前文件内容
3. 即使当前未选择文件，刷新按钮也保持可用
4. 如果用户在预览区选中了文字，前端立即把这次选中结果保存成状态
5. 即使用户之后切到输入框打字，这个“已选中”状态仍然保留
6. 如果存在有效选区状态，顶部状态区分两行显示：
   - 第一行显示 `已选中第 x 行到第 y 行`
   - 第二行显示被选中文字的缩略内容
7. 这里的行号按选中文本中的换行数近似估算
8. 缩略内容只显示前一小段文本，超出后截断
9. 提供 `清除` 按钮，用户可手动清空当前选中状态，并同时清空当前正在预览的文件路径
10. 不做正文内持久高亮，预览正文仍保持普通 Markdown 显示

消息发送附带信息：

1. 发送用户消息时，额外附带当前预览文件相对路径
2. 如果预览区存在有效选中文本状态，也一并附带
3. 后端收到后，在本次调用 LLM 前临时包进一段 `<系统提示>...</系统提示>`，并把用户消息放在最后
4. 这段附加上下文不进入历史消息，也不落盘

## 布局方案

右栏从当前“窄按钮列”改成“可拖拽宽度的侧边面板”。

建议结构：

- 顶部保留原有 `Token Stats` 和 `Knowledge Base`
- 中间是固定 3 列文件选择器
- 底部是文件预览区

宽度规则建议：

- 默认宽度：`380px`
- 最小宽度：`280px`
- 最大宽度：`720px`
- 用户拖拽后实时生效
- 当前宽度写入 `localStorage`，刷新页面后保留

右栏内部再分成上下两块：

- 上半部分：文件选择器
- 下半部分：文件预览区
- 预览区高度大于文件选择器

布局原则：

- 调整右栏宽度时，不影响主聊天区可用性
- 文件选择区和预览区各自可滚动
- 预览区优先保证阅读空间

## 文件选择器方案

不做传统树形缩进，而是做“固定 3 列路径浏览”。

列定义：

1. 第一列：根目录列，固定为 `knowledge_base` / `inputs`
2. 第二列：当前根目录下的直接子项
3. 第三列：当前第二列选中目录的直接子项

深入规则：

- 当用户继续进入更深层目录时，不再新增第 4 列
- 第三列始终复用为“当前深入层级的内容”
- 回退时按路径栈回到上一级
- 也可以直接点击上层列项，立即切换下级内容
- 每次切换到新的一列内容时，不自动打开任何文件
- 只有用户明确点击某个文件时，才更新预览区
- 当第三列没有更深层历史可退时，返回按钮禁用

这样做的好处：

- UI 宽度更可控
- 不会因为目录太深把右栏撑得很乱
- 比单列树更清楚，也比无限加列更稳定

目录项和文件项建议区分样式：

- 目录显示文件夹图标或目录标识
- 文件显示文件图标或文件标识
- 当前选中项高亮

文件：

- `src/server/public/index.html`

计划改动：

1. 把当前右侧按钮区改成可调宽度的右栏容器
2. 增加拖拽分隔条
3. 增加固定 3 列文件选择器 UI

6. 增加刷新按钮
7. 增加清除选中按钮
8. 复用现有 `renderMarkdown()` 渲染 Markdown 内容

前端状态大致需要：

- 右栏宽度
- 当前根目录
- 当前路径栈
- 第二维列表内容
- 第三维列表内容
- 当前选中文件路径
- 当前预览内容
- 当前预览区选中文本状态
- 当前预览区选中文本行数

## 服务端实现计划

文件：

- `src/server/index.ts`

建议新增两个接口：

1. 目录读取接口
   - 根据路径返回该层级下的直接子项
   - 子项包含类型信息：目录或文件
   - 仅允许访问 `knowledge_base` 和 `inputs`

2. 文件内容接口
   - 根据路径返回文件内容
   - 仅允许读取受控目录中的文件

另外需要扩展聊天消息接口：

3. 发送消息时附带预览信息
   - 当前预览文件相对路径
   - 当前选中文本（如果有）

接口职责建议：

- 目录接口只返回当前层级，不递归整棵树
- 文件接口返回完整文本内容
- 服务端负责做路径校验，避免越权访问

## 刷新方案

第一版采用手动刷新：

- 预览头部提供一个 `刷新` 按钮
- 点击后重新请求文档结构
- 如果当前已有选中文件且该文件仍然有效，再额外重新请求当前文件内容
- 即使当前没有选中文件，按钮也保持可用
- 刷新后直接清空当前选中状态
- 如果刷新后当前选中文件已不存在，则按默认规则回退
- 当前文件失效时，不自动打开新的文件预览

这样做的好处：

- 实现简单
- 行为可预期
- 不会因为模型持续写文件而频繁打断阅读

## 消息附带信息方案

第一版目标是把这两项信息稳定传到后端，并只在本次调用 LLM 时作为临时附加上下文使用。

发送用户消息时，前端附带两类可选信息：

1. 当前正在预览的文件
   - 例如：`knowledge_base/docs/guide.md`

2. 当前预览区选中的文字
   - 仅当选区发生在预览区内部时才附带
   - 没有选中文本时可为空
   - 前端按选中文本中的换行数估算并显示“已选中第 x 行到第 y 行”

建议的前端处理：

- 维护 `currentPreviewPath`
- 维护 `selectedPreviewText`
- 维护 `selectedPreviewLineCount`
- 维护 `selectedPreviewSummary`
- 仅当选区发生在预览区内部时更新这些状态值
- 用户切到输入框继续输入时，不清空这些状态值
- 浏览器原生选区消失时，不自动清空这些状态值
- 在预览区内部单击且没有新的有效选区时，清空这些状态值
- 文件切换时清空这些状态值
- 刷新后清空这些状态值
- 用户点击 `清除` 时清空这些状态值，并清空当前预览文件路径
- 在点击发送时，把以下字段一起发给后端：
  - `previewPath`
  - `selectedPreviewText`

建议的后端处理：

- 扩展 `chatMessage` 请求结构
- 收到后记录“本次实际发给 LLM 的最新 user message”日志，便于排查
- 在真正发给 LLM 前，临时组装成“系统提示在前，用户消息在后”的单次请求文本
- 不修改历史消息数组
- 不把这段附加上下文写入长期会话状态
- 不落盘

建议拼接格式：

```md
<系统提示>
当前用户正在浏览文件：{previewPath}

选中内容：
```text
{selectedPreviewText}
```

注意：行号基于估算，读取时可以前后多读两行。
用户正在浏览的文件不一定与当前对话直接相关，请结合上下文判断是否需要参考。

</系统提示>
用户消息：{user message}
```

拼接规则：

- 只用于本次请求发给 LLM 的最新输入
- 如果 `previewPath` 和 `selectedPreviewText` 都为空，则不拼接
- 如果只有 `previewPath`，则只保留文件路径提示
- 如果有 `selectedPreviewText`，则一起放进 `<系统提示>` 中
- “注意：行号基于估算，读取时可以前后多读两行。” 只在存在 `selectedPreviewText` 时拼接

状态保持规则：

- 选区一旦在预览区内被成功记录，就作为前端持久状态保留
- 用户后续点击输入框、输入消息，不影响这份状态
- 发送消息后默认不自动清空，便于连续追问同一段内容

状态清零规则：

- 用户点击 `清除`：清空选中状态，并清空当前预览文件路径
- 用户在预览区内部单击，且这次没有形成新的有效选区
- 用户切换到另一个文件
- 用户点击 `刷新`
- 刷新后当前文件失效并发生回退

## 选中文本可行性

“文件内容选中过后，去输入框打字就会被 deselect，怎么办？”答案是：不要依赖发送瞬间的浏览器原生选区，而是把选中结果保存成前端状态。

方案 A：

- 前端在预览区发生有效选择时，立即读取并保存当前选中文本
- 只在选区位于预览区 DOM 内时才采集
- 后续发送消息时直接使用这份状态值，而不是重新读取浏览器当前 selection
- 顶部显示状态提示：
  - `已选中第 x 行到第 y 行`
  - 被选中文字的缩略内容
- 不做正文内持久可视高亮

边界说明：

- 如果只是点击光标、没有实际选中文本，就拿不到内容
- 如果选区跨出了预览区，不建议采集
- Markdown 渲染后拿到的是“渲染后的可见文本”，不是原始 Markdown 源码片段
- 代码块、列表、标题通常都能选中，但格式信息会部分丢失
- “已选中第 x 行到第 y 行”按选中文本中的换行数近似估算，不按视觉换行计算
- 缩略内容取纯文本前一小段，不保留复杂格式
- 不必追求和原始 Markdown 行号完全一致

因此第一版建议：

- 先支持“读取预览区当前选中的纯文本”
- 前端维护一份当前选区状态，发送时直接使用这个状态值
- 不做复杂的原文定位
- 不做正文内持久可视高亮
- 只做顶部状态提示和缩略内容展示

## 可调宽度方案

右栏宽度通过拖拽分隔条调整。

建议交互：

1. 鼠标按下分隔条开始拖拽
2. 横向移动时实时更新右栏宽度
3. 松开鼠标结束拖拽
4. 宽度限制在最小值和最大值之间
5. 最终宽度写入 `localStorage`

实现上可以用原生鼠标事件：

- `mousedown`
- `mousemove`
- `mouseup`

如果后面需要，也可以再补双击恢复默认宽度，但第一版可以先不做。

## 第一版范围

第一版建议只做：

- 可拖拽调节右栏宽度
- 顶层固定 `knowledge_base` / `inputs`
- 固定 3 列目录浏览
- 深入时复用最后一列
- 回退
- 文件内容预览
- Markdown 渲染
- 手动刷新
- 清除选中状态
- 顶部选中状态提示
- 顶部选中文字缩略展示
- 发送消息时附带当前预览文件路径
- 发送消息时附带预览区选中文本

先不做：

- 搜索
- 自动刷新
- 多标签预览
- 历史浏览记录
- 文内全文高亮
- 正文内持久可视高亮
- 图片 / PDF 等特殊文件预览

## 验收标准

做到以下几点就算第一版完成：

1. 右栏可以通过拖拽调整宽度
2. 第一列固定显示 `knowledge_base` 和 `inputs`
3. 第二列和第三列能按当前选择动态更新
4. 继续深入目录时，第三列会复用显示更深层内容
5. 可以回退到上一级
6. 点击文件后，下方能显示预览内容
7. 非文本文件会显示“暂不支持预览”
8. Markdown 文件能正确渲染
9. 点击刷新按钮后，能重新加载文档结构
10. 发送用户消息时，后端能收到当前预览文件相对路径
11. 在预览区选中文本后发送消息时，后端能收到该选中文本
12. 后端会记录本次实际发给 LLM 的最新 user message 日志，便于核对附加上下文拼接结果
13. 如果当前已有选中文件且该文件仍然有效，点击刷新后会重新加载该文件内容
14. 刷新后会清空当前选中状态
15. 如果刷新后当前文件失效，会按默认规则回退，且不自动打开新文件
16. 预览区不做正文内持久可视高亮，只在顶部显示选中状态提示
17. 顶部状态提示会同时显示估算行区间和选中文字缩略内容
