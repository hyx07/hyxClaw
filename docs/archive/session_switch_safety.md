# 会话切换安全方案

## 目标

在完整的多会话并发前端支持实现之前，当前方案保证用户可以在模型生成过程中安全切换会话：

- 切换到新会话后，输入框立即可用
- 后台会话的流式输出不会渲染到当前会话
- 快速切换后立刻发送消息时，消息会写入用户当前选择的会话
- 后台会话生成失败时，错误不会污染当前正在查看的会话

本方案不追踪多个会话的实时生成状态，只保证“当前正在查看的会话”状态安全、可用。

## 核心原则

### 消息归属以 sessionId 为准

所有与具体会话相关的聊天消息都携带 `sessionId`。

前端发送用户消息时，`chatMessage` 携带当前会话 ID：

```js
{
  type: "chatMessage",
  sessionId: currentSessionId,
  content,
  enableThinking
}
```

后端处理聊天请求时，以 `chatMessage.sessionId` 作为消息归属，不依赖 WebSocket 连接上可能滞后的旧会话状态。

这样可以避免一种竞态：用户刚切换到会话 B 后立刻发送消息，而后端还没处理完 `joinSession(B)`，导致消息误写入旧会话 A。

### WebSocket 连接仍有当前订阅会话

WebSocket 连接上的 `client.sessionId` 表示该连接当前订阅哪个会话的实时下行事件。

当用户进入某个会话时，前端发送：

```js
{
  type: "joinSession",
  sessionId
}
```

后端会把该连接绑定到这个会话，并返回该会话的历史消息。

当用户发送 `chatMessage` 时，后端也会同步把该连接的当前订阅会话设置为 `chatMessage.sessionId`，确保后续的 `chatStart`、`chatChunk`、`toolCall`、`chatEnd` 可以正常返回给当前连接。

### 下行事件按会话隔离

后端所有与生成过程相关的下行事件都携带 `sessionId`：

- `chatStart`
- `chatReasoning`
- `toolCall`
- `toolResult`
- `chatChunk`
- `chatEnd`
- 带归属的 `error`

后端广播这些事件时，只发送给当前订阅该 `sessionId` 的连接。

因此，如果用户在会话 A 生成过程中切换到会话 B：

- A 的后续生成仍属于 A
- A 的后续事件不会发送给已经切到 B 的连接
- B 不会显示 A 的流式内容

### 前端只处理当前会话事件

前端收到 WebSocket 消息后，会检查事件是否属于当前会话。

对于流式事件，如果 `msg.sessionId` 存在且不等于 `currentSessionId`，前端直接忽略。

带 `sessionId` 的错误也只在对应会话中显示；不带 `sessionId` 的错误视为全局错误，可以显示在当前界面。

这层前端守卫用于防止异常情况下的跨会话渲染，也让消息处理逻辑更明确。

## 会话切换行为

### 切换会话时放弃旧的前端流式状态

当用户选择另一个会话时，前端会立即把本地流式状态重置：

- 当前不再视为 streaming
- 当前流式 assistant 气泡清空
- 当前 thinking 块清空
- 待填充的工具调用块清空
- 输入框恢复可用
- token 展示按新会话状态重置
- 新会话历史通过 `sessionLoaded` 重新渲染

这意味着：切换会话不是“暂停并保留前端流式 UI”，而是“放弃对旧会话当前生成过程的前端跟踪”。

旧会话的生成不会因此停止；它仍在后端继续执行并写入对应会话文件。

### 切回后台会话时重新加载历史

如果会话 A 正在后台生成，用户切到 B 后再切回 A，前端会重新 `joinSession(A)`。

后端会从 `$HYXCLAW_DATA_DIR/conversations/<sessionId>.jsonl` 读取 A 的最新历史，并通过 `sessionLoaded` 返回给前端。

因此，后台生成完成后的结果可以在切回会话时看到，但不会在用户停留于其他会话时实时显示。

## 典型场景

### 场景一：A 生成中切到 B

过程：

1. 用户在会话 A 发送消息
2. A 开始流式生成
3. 用户切换到会话 B

结果：

- B 的输入框立即可用
- A 的流式内容不会渲染到 B
- A 的后续生成继续在后端运行
- A 的最终结果继续写入 A 的 `.jsonl` 文件
- 用户切回 A 后，通过重新加载历史查看结果

### 场景二：切到 B 后立刻发送

过程：

1. 用户从 A 切换到 B
2. 用户马上在 B 输入并发送消息

结果：

- 前端发送的 `chatMessage` 携带 `sessionId: B`
- 后端按消息里的 `sessionId` 处理
- 该消息写入 B
- 不会因为 `joinSession(B)` 尚未处理完成而误写入 A

### 场景三：后台会话生成失败

过程：

1. A 正在后台生成
2. 用户已经切到 B
3. A 生成失败

结果：

- A 的错误事件携带 `sessionId: A`
- 当前 B 页面不会显示 A 的错误
- 如果用户切回 A，可以通过历史或后续机制查看对应会话状态

## 当前边界

当前方案不提供真正的多会话并发前端体验。

以下能力暂不支持：

- 同时显示多个会话的流式生成进度
- 后台会话完成后主动刷新侧边栏消息数
- 每个会话维护独立的前端 streaming state
- 切换回后台会话后继续接上未完成的流式 UI
- 在当前页面弹出后台会话完成或失败通知

这些能力可以留到后续“多会话并发”阶段实现。

## 数据持久化关系

会话文件仍然是最终事实来源。

生成过程中，后端会把正式进入会话历史的消息写入：

```text
$HYXCLAW_DATA_DIR/conversations/<sessionId>.jsonl
```

因此，即使前端切走、刷新或重新进入会话，只要后端已经完成落盘，历史记录仍可以通过 `sessionLoaded` 重新加载。
